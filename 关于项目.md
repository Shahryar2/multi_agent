### 项目核心概念
普通的 RAG 只能回答“苹果公司去年的股价是多少”。
而这个系统要解决的问题是：“**请帮我写一份关于2025年东南亚电动车市场的深度分析报告，包含竞争对手分析、政策风险和未来趋势预测。**”

这需要系统具备：**拆解任务 -> 自主上网搜索 -> 阅读大量网页/PDF -> 过滤噪音 -> 交叉验证 -> 撰写长文 -> 自我反思修改** 的能力。

---

### 为什么这个项目含金量极高？
1.  **避开了微调**：完全依靠 Prompt Engineering 和 Agent 架构设计。
2.  **极强的工程挑战**：
    *   **上下文管理**：搜索几十个网页，Token 肯定爆，如何做信息压缩（Map-Reduce）？
    *   **幻觉控制**：如何确保生成的报告每一句话都有据可查（Citations）？
    *   **死循环处理**：Agent 陷入搜索死循环怎么办？
3.  **当前最火的技术栈**：**LangGraph** (图导向的 Agent 编排) + **Multi-Agent** (多智能体协作)。

### 项目架构设计 (你的工作量所在)

你需要设计至少 4 个不同角色的智能体（Agent），并用 **LangGraph** 编排它们的状态流转：

#### 1. 规划者 (Planner Agent)
*   **职责**：接收用户模糊的指令，将其拆解为具体的搜索子任务（Sub-queries）。
*   **技术点**：思维链（CoT）Prompt，动态规划。
*   **输入**：“分析东南亚电动车市场”
*   **输出**：`["查询泰国EV补贴政策", "查询比亚迪在越南的销量", "查询印尼镍矿出口限制"]`

#### 2. 研究员 (Researcher Agent) - *最核心*
*   **职责**：执行搜索，读取网页，提取关键信息。
*   **工具**：`Tavily` 或 `Google Search API` (搜索)，`BeautifulSoup/Playwright` (爬虫)。
*   **技术难点**：
    *   **递归搜索**：如果第一层搜索结果不够好，它需要自主决定是否进行“追问”或“换个关键词搜”。
    *   **内容清洗**：网页全是广告和废话，需要用 LLM 提取核心事实（Fact Extraction）。

#### 3. 审稿人 (Reviewer/Critic Agent)
*   **职责**：检查研究员找回来的信息是否足够回答 Planner 的问题。
*   **技术点**：**自我反思 (Self-Reflection)**。如果信息缺失，它会把任务打回给研究员，要求“补充数据”。

#### 4. 撰稿人 (Writer Agent)
*   **职责**：将所有碎片化的事实，整合成一篇逻辑通顺、带有 Markdown 格式和引用标注（Footnotes）的万字长文。
*   **技术点**：长文本生成（Long-context generation），避免重复。

### 推荐技术栈

*   **编排框架**: **LangGraph** (必选，这是目前最能体现你对 Agent 状态机理解的框架，比 LangChain Chain 更有深度)。
*   **LLM**: GPT-4o 或 Claude 3.5 Sonnet (API调用，智能体对模型智商要求高)。
*   **搜索工具**: Tavily API (专为 Agent 设计的搜索引擎)。
*   **向量库**: ChromaDB / Faiss (用于暂存搜索到的网页片段，做去重和检索)。
*   **前端**: Streamlit 或 Next.js (需要展示实时的 Agent 思考过程，比如“正在阅读网页 A...”，“正在修正大纲...”)。

我要先给出建议结论，然后列出每种方案的利弊、面试展示价值和推荐的短/中/长期实施步骤。

- 结论（优先级）: 立刻用“截断+限数”快速可用；同时把“Map‑Reduce 摘要 + 向量库检索”作为中长期方案实现并在面试中展示，这是最能体现工程与研究能力的方案。

理由与对比
- 快速截断（短期）
  - **优点**: 实现最简单，立刻解决 token 超限和请求失败；低开发成本，适合调试与演示。  
  - **缺点**: 可能丢失细节、降低引用覆盖率，长期会造成质量/可信度问题。  
  - **适用场景**: 本地开发、快速 demo、CI 测试。  
- Map‑Reduce 摘要（中期）
  - **优点**: 保存关键信息、显著降 token，生成质量与可追溯性更好；能做章节级生成并保持上下文一致性。  
  - **缺点**: 需要实现 chunking、并发摘要、合并策略，调用更多 LLM 请求（成本上升但可控）。  
  - **适用场景**: 真实研报生成、保证可解释性与引用的场景。  
- 向量库 + 检索增强（长期）
  - **优点**: 可扩展、支持历史知识复用、只检索相关片段、方便去重与检索命中追踪（metadata 可做 citations）。  
  - **缺点**: 需引入 Chroma/FAISS/PGVector，设计 embedding 与检索策略，有运维成本。  
  - **适用场景**: 持续运行的产品、需要积累与复用大量文献/片段。

面试角度（哪个最加分）
- 最有价值：**Map‑Reduce + 向量库（结合 citations）**。这能同时展示：
  - 系统设计能力（分层抽象、可伸缩性）；
  - 工程实现（并发、错误处理、持久化）；
  - 产品感（引用溯源、Human-in-the-loop、可重现性）。
- 快速截断适合作为演示中的“快速修复”案例，但单独展示价值有限。

```

**数据流向图**：
```
ROUTER_PROMPT + 任务 → LLM → 分类结果
                         ↓

```python
{
    "documents": final_docs_for_state,         # 字典列表，前端直接展示
    "current_step_index": len(plan),           # 更新进度
    "plan": plan,                              # 更新每个任务的 status/result/doc_ids
    "bg_investigation": all_raw_results,       # 原始搜索结果（调试用）
    "search_data": all_raw_results             # 前端搜索卡片数据（包含图片）
}
```

**数据流向图**：
```
Plan [pending, pending, pending]
    ↓
ThreadPoolExecutor (5 workers)
    ↓
execute_single_task (x3 并行)
    ├─ 查询 + 搜索
    ├─ 转换为 Document
    ├─ 存入向量库
    ├─ 总结
    └─ 返回 {success, results, docs, docs_ids, raw_results}
    ↓
results_map { idx: result }
    ↓
Plan [completed, completed, completed]
documents: [dict, dict, dict]
search_data: [raw, raw, raw]
```

### **5. writer_node（撰写节点）** ⭐ 核心节点

**目的**：基于搜索结果和计划，生成最终的研究报告

**输入状态**：
```python
{
    "task": "给我春节小众旅游攻略",
    "style": "professional",
    "plan": [
        {
            "description": "...",
            "result": "根据搜索，推荐...",           # research_node 的 step_summary
            "doc_ids": ["url1", "url2", ...]       # 该步骤的文档 URL
        },
        ...
    ],
    "documents": [
        {
            "id": "url1",
            "text": "...",
            "title": "...",
            "url": "url1"
        },
        ...
    ]
}
```

**执行流程**：

#### **5.1 构建计划上下文**
```python
plan_context = ""
for step in plan:
    plan_context += f"### 研究步骤:{step.get('description')}\n"
    plan_context += f"### 研究结论:{step.get('result','无结果')}\n\n"
# 例如：
# ### 研究步骤:搜索2025年春节期间人流量较少的目的地推荐
# ### 研究结论:根据搜索，推荐云南丽江、贵州铜仁...
```

#### **5.2 RAG 检索：精准召回**
```python
# 第一阶段：按 doc_ids 精准检索
all_step_ids = []
for step in plan:
    all_step_ids.extend(step.get("doc_ids", []))  # 收集所有搜索到的 URL

if all_step_ids:
    full_docs = vector_store.get_documents_by_ids(all_step_ids)
    # 返回存储在向量库中的完整文档
```

#### **5.3 RAG 检索：补充召回**
如果精准召回不足 5 篇文档，进行相似度检索补充：
```python
for step in plan:
    query = step.get("description")
    results = vector_store.similarity_search(query, k=4)
    # 基于向量相似度，找最相关的 4 篇
```

#### **5.4 后备方案：Token 截断**
如果向量库无结果（通常是向量库故障），回退到直接截断：
```python
availble_for_docs = MODEL_LIMIT - RESERVED_OUTPUT - SYSTEM_PROMPT_ESTIMATE - plan_tokens
retrieved_docs = term_document(documents, max_tokens=availble_for_docs)
# 直接按 Token 限制截断 documents 列表
```

#### **5.5 生成引用列表**
```python
citations = []
for idx, seg in enumerate(retrieved_docs, start=1):
    is_dict = isinstance(seg, dict)
    text_content = seg.get("text") if is_dict else seg.page_content
    title_content = seg.get("title") if is_dict else seg.metadata.get("title", "Untitled")
    url_content = seg.get("url") if is_dict else seg.metadata.get("source", "")
    doc_id = seg.get("id") if is_dict else seg.metadata.get("source", "unknown")

    citations.append({
        "index": idx,
        "id": doc_id,
        "title": title_content,
        "url": url_content,
        "snippet": text_content[:300],
        "full_text": text_content
    })
# 返回 [{"index": 1, "title": "...", "url": "...", "snippet": "..."}, ...]
```

#### **5.6 构建上下文字符串**
```python
docs_context = "\n\n".join([
    f"[{c['index']}] Title: {c['title']}\nContent: {c['full_text']}\nSource: {c['url']}" 
    for c in citations
])
# 例如：
# [1] Title: 云南丽江旅游指南
# Content: 丽江古城是国家级历史文化名城...
# Source: https://example.com/lijiang
```

#### **5.7 调用 LLM 生成报告**
```python
full_prompt = WRITER_PROMPT.format(
    task=task,                  # "给我春节小众旅游攻略"
    style=style,                # "professional"
    plan_context=plan_context,  # 研究步骤 + 结论
    docs_context=docs_context   # 完整参考文献上下文
)

response = llm.invoke([HumanMessage(content=full_prompt)])
final_draft = response.content
# 例如：生成一篇 2000 字的旅游攻略报告
```

#### **5.8 追加引用列表**
```python
citations_footer = "\n\n## 引用列表\n" + "\n".join(
    f"[{c['index']}] {c['title']} — {c['url']}" for c in citations
)
return {
    "draft": final_draft + citations_footer,
    "citations": citations
}
```

**输出状态**：
```python
{
    "draft": "# 春节小众旅游攻略\n\n## 推荐目的地\n云南丽江...\n\n## 引用列表\n[1] ...",
    "citations": [
        {"index": 1, "title": "...", "url": "...", "snippet": "..."},
        ...
    ]
}
```

### **6. verifier_node（验证节点）**

**目的**：检查报告中的引用索引是否完整（防止幻觉）

**输入状态**：
```python
{
    "draft": "# 报告\n参考文献[1][2][3]...\n## 引用列表\n[1] ... [2] ...",
    "citations": [
        {"index": 1, "title": "..."},
        {"index": 2, "title": "..."}
    ]
}
```

**执行逻辑**：
```python
# 提取报告中出现的引用索引
main_indices = draft.split("## 引用列表")[0]
used_indices = set(re.findall(r"\[(\d+)\]", main_indices))
# 例如：{1, 2, 3}

# 检查引用列表中实际存在的索引
existing_indices = {c['index'] for c in citations}
# 例如：{1, 2}

# 找出缺失的引用
missing_indices = used_indices - existing_indices
# 例如：{3}

if missing_indices and revision_number < max_revisions:
    # 缺失引用，且还有修改机会，触发重写
    return {
        "next": "writer",
        "revision_number": revision_number + 1
    }
else:
    # 引用完整或达到修改上限，进入审核阶段
    return {"next": "reviewer"}
```

### **7. reviewer_node（审核节点）**

**目的**：由 LLM 审核报告的质量，决定是否通过或需要修改

**输入状态**：
```python
{
    "task": "给我春节小众旅游攻略",
    "draft": "# 春节小众旅游攻略\n...",
    "revision_number": 0
}
```

**执行逻辑**：
```python
# 调用 Reviewer 提示词，让 LLM 评估报告
llm.invoke([HumanMessage(content=REVIEVER_PROMPT.format(
    task=task,
    draft_segment=draft[:10000]  # 只发送前 10000 字（防止过长）
))])

# LLM 返回 JSON 格式的审核结果
review_data = {
    "status": "pass" 或 "fail",
    "reason": "通过原因或未通过原因",
    "missing": "缺失的内容（如果 status=fail）"
}
```

**流程决策**：
- **status=pass** → 工作流结束，返回最终报告
- **status=fail** → 回到 `planner_node`，基于 `review_data.missing` 调整计划，重新搜索和写作

**输出状态**：
```python
{
    "review": {
        "status": "pass" 或 "fail",
        "reason": "...",
        "missing": "..."
    },
    "revision_number": revision_number + 1
}
```
