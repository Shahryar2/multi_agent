### 项目核心概念
普通的 RAG 只能回答“苹果公司去年的股价是多少”。
而这个系统要解决的问题是：“**请帮我写一份关于2025年东南亚电动车市场的深度分析报告，包含竞争对手分析、政策风险和未来趋势预测。**”

这需要系统具备：**拆解任务 -> 自主上网搜索 -> 阅读大量网页/PDF -> 过滤噪音 -> 交叉验证 -> 撰写长文 -> 自我反思修改** 的能力。

---

### 为什么这个项目含金量极高？
1.  **避开了微调**：完全依靠 Prompt Engineering 和 Agent 架构设计。
2.  **极强的工程挑战**：
    *   **上下文管理**：搜索几十个网页，Token 肯定爆，如何做信息压缩（Map-Reduce）？
    *   **幻觉控制**：如何确保生成的报告每一句话都有据可查（Citations）？
    *   **死循环处理**：Agent 陷入搜索死循环怎么办？
3.  **当前最火的技术栈**：**LangGraph** (图导向的 Agent 编排) + **Multi-Agent** (多智能体协作)。

---

### 项目架构设计 (你的工作量所在)

你需要设计至少 4 个不同角色的智能体（Agent），并用 **LangGraph** 编排它们的状态流转：

#### 1. 规划者 (Planner Agent)
*   **职责**：接收用户模糊的指令，将其拆解为具体的搜索子任务（Sub-queries）。
*   **技术点**：思维链（CoT）Prompt，动态规划。
*   **输入**：“分析东南亚电动车市场”
*   **输出**：`["查询泰国EV补贴政策", "查询比亚迪在越南的销量", "查询印尼镍矿出口限制"]`

#### 2. 研究员 (Researcher Agent) - *最核心*
*   **职责**：执行搜索，读取网页，提取关键信息。
*   **工具**：`Tavily` 或 `Google Search API` (搜索)，`BeautifulSoup/Playwright` (爬虫)。
*   **技术难点**：
    *   **递归搜索**：如果第一层搜索结果不够好，它需要自主决定是否进行“追问”或“换个关键词搜”。
    *   **内容清洗**：网页全是广告和废话，需要用 LLM 提取核心事实（Fact Extraction）。

#### 3. 审稿人 (Reviewer/Critic Agent)
*   **职责**：检查研究员找回来的信息是否足够回答 Planner 的问题。
*   **技术点**：**自我反思 (Self-Reflection)**。如果信息缺失，它会把任务打回给研究员，要求“补充数据”。

#### 4. 撰稿人 (Writer Agent)
*   **职责**：将所有碎片化的事实，整合成一篇逻辑通顺、带有 Markdown 格式和引用标注（Footnotes）的万字长文。
*   **技术点**：长文本生成（Long-context generation），避免重复。

---

### 推荐技术栈

*   **编排框架**: **LangGraph** (必选，这是目前最能体现你对 Agent 状态机理解的框架，比 LangChain Chain 更有深度)。
*   **LLM**: GPT-4o 或 Claude 3.5 Sonnet (API调用，智能体对模型智商要求高)。
*   **搜索工具**: Tavily API (专为 Agent 设计的搜索引擎)。
*   **向量库**: ChromaDB / Faiss (用于暂存搜索到的网页片段，做去重和检索)。
*   **前端**: Streamlit 或 Next.js (需要展示实时的 Agent 思考过程，比如“正在阅读网页 A...”，“正在修正大纲...”)。

### 面试时的“杀手锏”功能 (加分项)

为了让这个项目在面试中脱颖而出，你必须实现以下 2-3 个高级特性：

1.  **人机协同 (Human-in-the-loop)**：
    *   在 Planner 拆解完任务后，**暂停**运行，弹窗询问用户：“我计划从这 3 个方向进行调研，您觉得需要修改吗？”
    *   这展示了你对 LangGraph **中断（Interrupt）**和**状态持久化（Checkpointer）**机制的掌握。

2.  **真实引用溯源 (Grounding with Citations)**：
    *   报告中的每一段话，鼠标悬停都要能显示来源 URL。
    *   这证明你解决了 LLM 最大的痛点——胡说八道。

3.  **动态资源调度**：
    *   实现**并行搜索**。Planner 拆出 5 个子任务，同时启动 5 个 Researcher 线程并行抓取，最后汇总。
    *   这展示了你的**并发编程**能力。

我要先给出建议结论，然后列出每种方案的利弊、面试展示价值和推荐的短/中/长期实施步骤。

- 结论（优先级）: 立刻用“截断+限数”快速可用；同时把“Map‑Reduce 摘要 + 向量库检索”作为中长期方案实现并在面试中展示，这是最能体现工程与研究能力的方案。

理由与对比
- 快速截断（短期）
  - **优点**: 实现最简单，立刻解决 token 超限和请求失败；低开发成本，适合调试与演示。  
  - **缺点**: 可能丢失细节、降低引用覆盖率，长期会造成质量/可信度问题。  
  - **适用场景**: 本地开发、快速 demo、CI 测试。  
- Map‑Reduce 摘要（中期）
  - **优点**: 保存关键信息、显著降 token，生成质量与可追溯性更好；能做章节级生成并保持上下文一致性。  
  - **缺点**: 需要实现 chunking、并发摘要、合并策略，调用更多 LLM 请求（成本上升但可控）。  
  - **适用场景**: 真实研报生成、保证可解释性与引用的场景。  
- 向量库 + 检索增强（长期）
  - **优点**: 可扩展、支持历史知识复用、只检索相关片段、方便去重与检索命中追踪（metadata 可做 citations）。  
  - **缺点**: 需引入 Chroma/FAISS/PGVector，设计 embedding 与检索策略，有运维成本。  
  - **适用场景**: 持续运行的产品、需要积累与复用大量文献/片段。

面试角度（哪个最加分）
- 最有价值：**Map‑Reduce + 向量库（结合 citations）**。这能同时展示：
  - 系统设计能力（分层抽象、可伸缩性）；
  - 工程实现（并发、错误处理、持久化）；
  - 产品感（引用溯源、Human-in-the-loop、可重现性）。
- 快速截断适合作为演示中的“快速修复”案例，但单独展示价值有限。

短期/中期实施建议（具体步骤）
1. 立刻：在 `writer_node` 使用 `_condense_documents`（每段截断 + 最多 N 段）避免超限，快速恢复可跑通 demo。  
2. 中期（3-7 天）：实现 `utils/summarizer.py`：
   - `map_summarize(segment) -> short_summary`（并行对 chunk 做短摘要）；
   - `reduce_summaries(list) -> merged_summary`（合并摘要并生成写作上下文）；  
   - 把 `writer_node` 改为只喂入 `merged_summary + top-k citations`。  
3. 长期（后续）：引入向量库（Chroma/FAISS/PGVector）：
   - 在 `research_node` 将原始 segments 写入向量库并记录 `id,url,title,metadata`；
   - 写作时基于大纲/段落检索 top-k 片段喂给 LLM，结合 map-reduce 做最终生成与精确 citations。

- 请求成功率、平均 token 消耗、生成延迟、引用覆盖率（报告中断言被 source 支持的比率）。用这些指标对比“截断 vs map-reduce vs retriever”。
